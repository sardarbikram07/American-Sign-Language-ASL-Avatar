 \textbf{ABSTRACT}
 \addcontentsline{toc}{chapter}{Abstract}
\end{center}



\vspace{0.8cm}

American Sign Language (ASL) is a complete, natural language that exhibits the same linguistic complexities as spoken languages, including its own syntax, morphology, and grammar that significantly differ from English. This project presents an open-source American Sign Language fingerspell recognition and semantic pose retrieval system designed to facilitate communication between ASL signers and individuals who do not understand ASL. The system provides two main functionalities: a recognition module for translating ASL fingerspelling into spoken English and a production module for converting spoken English into ASL pose sequences.

The recognition component utilizes Google MediaPipe Hand Landmark model combined with a custom PointNet neural network architecture to achieve real-time fingerspelling recognition. The system captures 21 3D hand landmarks per frame and processes them through a trained PointNet model capable of classifying ASL alphabets with high accuracy. The production component employs a text-to-gloss translation pipeline followed by semantic search using sentence transformers to retrieve corresponding ASL poses from a comprehensive database of over 9,000 ASL signs.

The complete system is implemented as a WebSocket API that can be accessed through a user-friendly web interface built with React and ThreeJS. The interface enables real-time two-way communication between ASL and English, making it accessible for users in various environments with different lighting conditions, skin tones, hand sizes, and camera distances. All components are open-source and designed to be modular, allowing for independent usage of recognition and production modules.

\chapter{INTRODUCTION}
\section{Overview}

American Sign Language (ASL) is a complete, natural language that exhibits the same linguistic complexities as spoken languages, including its own syntax, morphology, and grammar that significantly differ from English. ASL is the primary language of many individuals in North America, and it is used by millions of people worldwide. Despite its prevalence, there remains a significant lack of resources and tools available to the public to facilitate communication between ASL and spoken language users.

The idea that ASL is merely a visual representation of English is a widespread misconception. In reality, ASL is a distinct language with its own syntax, grammar, and cultural nuances that differ significantly from spoken English. Most existing direct translation tools are built with this misunderstanding, failing to capture the depth of ASL communication. Furthermore, these tools put an even larger burden on the Deaf individual, requiring them to accommodate and adapt to meet the needs of the hearing individual.

This project presents an open-source American Sign Language fingerspell recognition and semantic pose retrieval system designed to bridge the communication gap between ASL signers and hearing individuals who do not understand sign language. The system is specifically designed to respect and preserve ASL as the primary language while providing practical translation capabilities.

The developed system provides two main components that can be used independently or together:

\begin{itemize}
    \item \textbf{Recognition Module:} Ability to interpret ASL fingerspelling and express it as spoken English text using computer vision and machine learning
    \item \textbf{Production Module:} Ability to interpret spoken English and express it as ASL signing pose sequences through semantic search and animation
\end{itemize}

To serve as a stepping stone towards more real-world sign language translation systems, the system is designed to be modular and adaptable. The system is implemented through a WebSocket API, making it accessible for integration into various applications and platforms.

\section{Problem Formulation}

The traditional flow of conversation between a Deaf and hearing individual when an interpreter is not present typically involves writing or typing to communicate, or using Text-to-Speech (TTS) and Speech-to-Text (STT) tools that primarily assist the hearing individual. This process creates several challenges:

\begin{enumerate}
    \item \textbf{Language Translation Burden:} ASL signers must constantly translate their thoughts from ASL into English, which is mentally taxing and often results in loss of meaning and context.
    
    \item \textbf{Accommodation Requirement:} Most existing tools require Deaf individuals to accommodate hearing people's communication preferences rather than providing true bidirectional translation.
    
    \item \textbf{Limited Real-world Applicability:} Existing solutions often fail in varied environmental conditions such as different lighting, backgrounds, skin tones, and camera distances.
    
    \item \textbf{Lack of ASL Grammar Consideration:} Most tools treat ASL as visual English, ignoring its unique grammatical structure and five fundamental parameters (handshape, movement, location, orientation, and non-manual markers).
    
    \item \textbf{Accessibility Barriers:} Current solutions are often expensive, require specialized hardware, or are not readily available to the general public.
\end{enumerate}

The core problem this project addresses is the development of a real-time, accessible, and accurate bidirectional communication system that respects ASL as a complete language while enabling effective communication between ASL signers and hearing individuals.

\section{Objectives}

The primary objectives of this project are:

\begin{itemize}
\item \textbf{Real-time ASL Fingerspelling Recognition:} Develop a computer vision system capable of accurately recognizing ASL fingerspelling in real-time using MediaPipe hand landmarks and PointNet neural network architecture, achieving high accuracy across diverse environmental conditions.

\item \textbf{English-to-ASL Translation System:} Create a production module that converts spoken English text into ASL gloss and retrieves corresponding sign animations using semantic search, enabling hearing individuals to communicate in visual language.

\item \textbf{Robust Environmental Adaptability:} Design the system to function effectively under varying conditions including different lighting, backgrounds, skin tones, hand sizes, and distances from camera, ensuring accessibility in real-world scenarios.

\item \textbf{Modular and Open-Source Architecture:} Implement a modular system design that allows independent usage of recognition and production components, with complete open-source availability to encourage further development and research.

\item \textbf{User-Friendly Interface Development:} Create an intuitive web-based interface using modern technologies like React and ThreeJS that enables seamless real-time interaction between ASL and English communication.

\item \textbf{Scalable Database Implementation:} Establish a comprehensive ASL signs database with semantic search capabilities using sentence transformers and PostgreSQL with pgvector, enabling contextual sign retrieval even for words not directly present in the database.
\end{itemize}

\section{Motivation}

The motivation for this project stems from the recognition that communication barriers between the Deaf and hearing communities persist despite technological advances. Traditional translation tools often view Deafness as a disability to be accommodated rather than recognizing ASL as a rich, complete language deserving of preservation and respect.

For over eight years, attempts at learning multiple spoken languages revealed the unique challenges of visual language learning. ASL presents different paradigms - it's not just about words or pronunciation, but rather learning to fully express oneself through visual and spatial means. This project emerged from observations of the hurdles faced by the Deaf community in our hearing-centric society.

The goal is to eliminate the extra steps that Deaf people have to take because of societal hearing-centrism. From the typical four extra steps in hearing-Deaf communication (ASL thought → English translation → writing/typing → hearing person reads → speaks response → Deaf person reads English → translates to ASL), this tool aims to eliminate three steps to a sufficient degree, creating a more direct communication flow.

Furthermore, fingerspelling accounts for approximately 25-35\% of expressed ASL, making it a crucial component for any ASL translation system. By focusing on fingerspelling recognition and basic sign production, this project serves as a foundation for more comprehensive ASL translation systems.

The ultimate vision extends beyond technical achievement to social impact: creating technology that brings ASL and hearing communities together, encouraging more people to learn about ASL culture and language, and providing a foundation for further accessibility technology development.

\chapter{BACKGROUND AND LITERATURE REVIEW}

\section{Sign Language Processing Overview}

Sign Language Processing (SLP) is a field of research that focuses on the development of technologies that can understand and generate sign language. With the rise of novel deep learning and computer vision technology, the field of SLP has seen significant advancements in recent years. Sign language processing can be segmented into two main categories: recognition and production.

Sign language recognition focuses on understanding sign language, while sign language production focuses on the generation of sign language. Both areas present unique challenges and have seen various approaches in recent literature.

\section{Recognition Techniques}

There are multiple components to the reception of sign language, including fingerspell recognition, sign language detection, sign language segmentation, and sign language recognition. Each component serves a specific purpose in the overall goal of sign language translation.

\subsection{Fingerspell Recognition}

Fingerspell recognition involves recognizing individual ASL letters in videos or images. This is a crucial component as fingerspelling accounts for a significant portion of ASL communication.

Previous work by researchers has utilized Microsoft Kinect devices to capture real-time depth and appearance images for ASL fingerspelling recognition. They found that classification using both depth and appearance images outperformed classification using only depth images. Other researchers have used videos from uncontrolled environments to train fingerspell recognition models that utilized iterative attention mechanisms to improve performance.

Traditional approaches have relied on image classification using Convolutional Neural Networks (CNNs). However, these approaches face challenges in real-world scenarios due to variations in lighting, backgrounds, skin tones, and hand sizes.

\subsection{Sign Language Detection}

Sign language detection is the binary classification task of determining whether a given video features someone signing. This technology has applications in video conferencing software to spotlight signers when they communicate.

Research has utilized optical flow features from OpenPose human pose estimation to train binary classifiers, producing results of 87\%-91\% accuracy with per-frame inference times of only 350-3500 microseconds. Other approaches have used multi-layer recurrent neural networks and 2-stream convolutional neural networks to analyze video and motion data for sign language detection.

\subsection{Sign Language Translation}

Complete sign language translation remains one of the most challenging aspects of SLP. Some systems have developed LSTM models that could translate individual signs into text with accuracies around 84.2\%. These systems typically require large datasets of isolated signs from multiple signers.

\section{Production Techniques}

Sign language production systems typically follow a pipeline of text-to-gloss-to-pose-to-video translation. Each step in this pipeline presents unique challenges and opportunities for innovation.

\subsection{Text to Gloss}

Converting spoken English text to ASL gloss requires understanding the grammatical differences between English and ASL. Research has explored various methods including lemmatizers, rule-based word reordering and dropping systems, and neural machine translation models.

Some approaches have integrated syntactic information from dependency parsers into word embeddings to develop syntax-aware transformer models for text-to-gloss translation. These methods show promise in incorporating the unique grammatical structure of ASL.

\subsection{Gloss to Pose}

Converting ASL gloss to poses involves mapping sign representations to skeletal pose data. Traditional approaches have used lookup tables that map sign glosses to 2D skeletal poses. More advanced approaches have used MediaPipe Holistic to extract body pose estimations from sign language datasets and then stitch poses together to create continuous signing sequences.

\subsection{Pose to Video}

The final step involves generating realistic videos of people signing from pose data. Research has explored using Pix2Pix models and ControlNet models to generate videos, with ControlNet generally outperforming traditional approaches in terms of realism and quality.

\section{Technology Stack}

\subsection{MediaPipe Framework}

Google's MediaPipe is an open-source framework for building perception pipelines for processing perceptual data. For hand tracking, MediaPipe provides real-time hand landmark detection that can identify 21 3D landmarks per hand with high accuracy and efficiency.

The MediaPipe Hand Landmark model offers several advantages:
\begin{itemize}
    \item Real-time performance on mobile and web platforms
    \item Invariance to lighting conditions, backgrounds, and skin tones
    \item Precise 3D landmark coordinates
    \item Low computational requirements
\end{itemize}

\subsection{PointNet Architecture}

PointNet is a neural network architecture designed for processing point cloud data. Unlike traditional CNNs that work on grid-like structures, PointNet can directly consume point clouds and is invariant to permutations of the input points.

For ASL fingerspelling recognition, PointNet offers advantages over traditional image classification approaches:
\begin{itemize}
    \item Direct processing of 3D hand landmarks
    \item Invariance to hand orientation and position
    \item Efficiency in processing sparse point data
    \item Robustness to variations in hand size and distance from camera
\end{itemize}

\subsection{Semantic Search Technologies}

Modern natural language processing has enabled semantic search capabilities through sentence transformers and embedding techniques. For ASL production, semantic search allows finding contextually similar signs even when exact matches are not available in the database.

The use of sentence transformers like "all-MiniLM-L6-v2" enables encoding words and phrases into high-dimensional vector spaces where semantic similarity can be measured using cosine similarity. This approach significantly expands the effective vocabulary of the system beyond the explicitly stored signs.

\chapter{METHODOLOGY AND SYSTEM DESIGN}

\section{System Architecture}

The ASL fingerspelling recognition and English-to-ASL translation system follows a modular architecture consisting of two primary components: the Recognition Module and the Production Module. Both modules are integrated through a WebSocket API that enables real-time communication between client and server.

\begin{figure}[H]
    \centering
    \includegraphics[scale=0.8]{system_architecture.jpg}
    \caption{Overall System Architecture}
    \label{fig:system_architecture}
\end{figure}

The system architecture follows a client-server model where:
\begin{itemize}
    \item \textbf{Frontend Client:} React-based web application with real-time video processing and 3D animation capabilities
    \item \textbf{Backend Server:} Flask application with WebSocket support for real-time communication
    \item \textbf{Recognition Pipeline:} MediaPipe + PointNet for fingerspelling detection and classification
    \item \textbf{Production Pipeline:} Text-to-gloss translation + semantic pose retrieval
    \item \textbf{Database:} PostgreSQL with pgvector extension for semantic search capabilities
\end{itemize}

\section{Recognition Module Design}

The recognition module implements a three-stage pipeline for converting real-time video input into English text through ASL fingerspelling recognition.

\subsection{Stage 1: Hand Landmark Detection}

The first stage utilizes Google MediaPipe Hand Landmark model to detect and track hand positions in real-time video streams. MediaPipe processes each frame and extracts 21 3D landmarks for each detected hand, providing precise coordinate information for hand pose estimation.

Key features of this stage:
\begin{itemize}
    \item Real-time processing at 30+ FPS
    \item Automatic hand detection and tracking
    \item 3D coordinate extraction with confidence scores
    \item Handedness classification (left/right hand)
    \item Normalization of coordinates relative to hand bounds
\end{itemize}

The normalization process ensures that the model is invariant to distance from camera and hand size by converting absolute coordinates to relative positions within the hand bounding box.

\subsection{Stage 2: Fingerspelling Classification}

The second stage employs a trained PointNet neural network to classify the normalized hand landmarks into ASL alphabet characters. The PointNet architecture is particularly suitable for this task as it can directly process point cloud data and is invariant to point ordering.

\textbf{Model Architecture:}
The PointNet model consists of:
\begin{itemize}
    \item Input layer: 21 points × 3 coordinates = 63 input features
    \item Point-wise transformation layers with shared MLPs
    \item Max pooling for global feature extraction
    \item Classification head with 29 output classes (26 letters + space + delete + nothing)
\end{itemize}

\textbf{Training Process:}
The model was trained on a combined dataset of over 120,000 labeled ASL fingerspelling images from multiple Kaggle datasets. The training process involved:
\begin{enumerate}
    \item Hand landmark extraction from all training images using MediaPipe
    \item Coordinate normalization and data augmentation
    \item Model training with various hyperparameter configurations
    \item Performance evaluation using accuracy, precision, and recall metrics
\end{enumerate}

\subsection{Stage 3: Text Synthesis and Error Correction}

The third stage converts the sequence of classified letters into coherent English text through multiple refinement processes:

\textbf{Temporal Consistency:} The system ensures that a letter is classified consistently across multiple consecutive frames before adding it to the output text, reducing noise from momentary misclassifications.

\textbf{Misrecognition Correction:} Common misclassifications between similar letters (A-T, K-V, U-R, E-S) are corrected using geometric analysis of specific landmark relationships.

\textbf{Contextual Synthesis:} A BERT-based language model corrects spelling errors and provides contextual improvements to the synthesized text.

\section{Production Module Design}

The production module converts spoken English text into ASL sign sequences through a two-stage pipeline involving text-to-gloss translation and semantic pose retrieval.

\subsection{Text-to-Gloss Translation}

The first stage converts English text into ASL gloss format, which represents the grammatical structure of ASL. This process involves:

\begin{itemize}
    \item Grammatical analysis of input text
    \item Word reordering according to ASL syntax rules
    \item Removal of English articles and connectors not present in ASL
    \item Temporal marker insertion for ASL-specific grammatical elements
\end{itemize}

The system uses a combination of rule-based approaches and large language model assistance to ensure accurate gloss generation.

\subsection{Semantic Pose Retrieval}

The second stage retrieves corresponding ASL poses for each gloss term using semantic search. This approach enables the system to handle words not explicitly present in the database by finding contextually similar signs.

\textbf{Database Design:}
The PostgreSQL database with pgvector extension stores:
\begin{itemize}
    \item Over 9,000 ASL signs with corresponding English words
    \item 384-dimensional sentence transformer embeddings for each word
    \item Pose sequence data extracted from sign language videos using MediaPipe
    \item Animation timing and transition information
\end{itemize}

\textbf{Semantic Search Process:}
\begin{enumerate}
    \item Input word embedding generation using sentence transformers
    \item Cosine similarity calculation against database embeddings
    \item Retrieval of best matching sign with similarity threshold (>0.70)
    \item Fallback to fingerspelling animation if no suitable match found
\end{enumerate}

\subsection{Animation Generation}

The retrieved poses are synthesized into continuous ASL signing animations using ThreeJS for real-time 3D rendering. The animation system handles:
\begin{itemize}
    \item Smooth transitions between different signs
    \item Proper timing for natural signing rhythm
    \item 2D avatar rendering with anatomically correct proportions
    \item Real-time playback synchronized with text input
\end{itemize}

\section*{Appendix B: Model Architecture Details}

\textbf{PointNet Architecture Configuration:}
\begin{itemize}
    \item Input: 21 landmarks × 3 coordinates = 63 features
    \item Point transformation: [64, 64, 64] fully connected layers
    \item Feature transformation: [64, 128, 1024] fully connected layers
    \item Global max pooling: 1024-dimensional feature vector
    \item Classification head: [512, 256, 29] fully connected layers
    \item Activation: ReLU for hidden layers, Softmax for output
    \item Regularization: Dropout (0.3) and L2 regularization
\end{itemize}

\textbf{Database Schema:}
\begin{itemize}
    \item Signs table: id, word, gloss, embedding (vector), points (JSON)
    \item Metadata table: sign\_id, duration, complexity, region
    \item Performance logs: timestamp, accuracy, processing\_time
\end{itemize}
